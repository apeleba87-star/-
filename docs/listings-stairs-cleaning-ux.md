# 계단 청소 — 평수 대신 층/옵션 기준 처리 방안

## 1. 견적계산기 계단 청소 구조 (현재)

`lib/estimate-calc.ts` · `components/estimate/CleaningEstimateCalculator.tsx` 기준으로 계단 청소는 **평수가 아니라 아래 항목**으로 견적이 나갑니다.

| 항목 | 설명 | 견적 반영 |
|------|------|-----------|
| **층수** (floors) | 기본 4층 기준, 그 이상은 층당 추가 단가 | `stairs_base_monthly` + (층수−4)×`stairs_extra_per_floor` |
| **주 회수** (visitsPerWeek) | 주 1~3회 | 배율 적용 (`stairs_visit_multiplier[0~2]`) |
| **화장실** | 개수 + 월 단가 | `restroomCount × stairs_restroom_unit` |
| **분리수거** | 유무 + 월 단가 | 있으면 `stairs_recycle_monthly` |
| **엘리베이터** | 유무 + 월 단가 | 있으면 `stairs_elevator_monthly` |
| **외부 주차장** | 유무 + 월 단가 | 있으면 `stairs_parking_monthly` |
| **창틀 먼지** | 유무 + 월 단가 | 있으면 `stairs_window_monthly` |
| **사용자 추가 항목** | 이름 + 월액 | extraItems 합산 |

- **복도**는 현재 견적계산기 항목에 **없음**. “복도가 있는지”는 별도 옵션으로 추가하는 것이 좋음.

---

## 2. 사용자 요구와의 매칭

말씀하신 항목과 견적계산기·저장 구조를 맞추면 다음과 같습니다.

| 요구 항목 | 견적계산기 | 비고 |
|-----------|------------|------|
| 몇 층인지 | ✅ 층수 (floors) | 동일 |
| 주 몇 회인지 | ✅ 방문 빈도 (주 1~3회) | 견적은 1~3회, 현장거래는 1~7회 가능 |
| 분리수거가 있는지 | ✅ recycle (유무 + 단가) | 동일 |
| 복도가 있는지 | ❌ 없음 | **신규 옵션** 권장 (유무만 먼저 저장 가능) |
| 화장실 청소가 있는지 | ✅ restroomCount (개수 + 단가) | “있음/없음” 또는 “개수”로 매핑 가능 |

그래서 **계단 청소만** “평수/제곱미터 + 주회수 + 난이도”가 아니라,  
**층수 → 주 회수 → 화장실 · 분리수거 · 복도 등 옵션** 순으로 묶는 게 맞습니다.

---

## 3. 처리 방안 제안

### 3.1 현장 거래 글쓰기(Step 3) 분기

- **업무 종류 = 계단청소**일 때만 Step 3를 **계단 전용**으로 전환.
- 그 외(사무실청소, 식당청소 등)는 기존처럼 **평수/제곱미터, 주 회수, 난이도** 유지.

**계단 전용 Step 3 예시**

1. **층수** (필수) — 숫자 입력, 예: 4층 기준 ~ N층
2. **주 회수** (필수) — 1~7회 선택 (견적은 1~3회만 써도 됨)
3. **화장실 청소** — 없음 / 있음(개수 입력)
4. **분리수거** — 없음 / 있음
5. **복도** — 없음 / 있음 *(견적계산기에는 없으므로 유무만 저장 후 추후 단가 연동)*
6. **기타 옵션** (선택) — 엘리베이터, 주차장, 창틀 먼지 등  
   - 견적계산기와 동일한 항목을 체크박스 + 단가(선택)로 두면, 나중에 “예상 월액” 등 참고값 계산과도 맞출 수 있음.

소개 + “견적 확인 필요”인 경우는 현재처럼 평수/주회수/난이도 비활성화만 유지하고,  
**계단청소 선택 시**에는 “견적 확인 필요”여도 **층수·주회수·옵션(화장실/분리수거/복도 등)** 은 받는 쪽이 자연스러울 수 있음. (필수까지 할지는 정책 결정.)

### 3.2 DB 저장

- `listings`에는 현재 **계단 전용 컬럼이 없음** (area_pyeong, visits_per_week 만 존재).
- 선택지:
  - **A. 계단 전용 컬럼 추가**  
    - 예: `stairs_floors`, `stairs_restroom_count`, `stairs_has_recycle`, `stairs_has_corridor` 등  
    - 장점: 쿼리·집계·견적 연동이 단순.
  - **B. JSONB 한 컬럼**  
    - 예: `stairs_options`  
    - `{ floors, visits_per_week, restroom_count, recycle, corridor, elevator, parking, window }`  
    - 장점: 스키마 변경 최소, 항목 추가 유연.

견적계산기와 동일한 단가 구조를 쓰려면, **옵션별 유무 + (선택) 단가**를 저장할 수 있게 설계하는 것이 좋습니다.

### 3.3 견적계산기와의 정합성

- 글쓰기에서 **계단청소** 선택 시 입력한 값(층수, 주회수, 화장실, 분리수거, 복도 등)을  
  `StairsInput` 형태로 맞추어 두면:
  - 나중에 **listing 상세/편집**에서 “예상 월액”을 `calcStairs(..., config)`로 계산해 보여줄 수 있고,
  - 견적계산기 페이지와 **동일한 단가·로직**을 재사용할 수 있습니다.
- **복도**는 견적계산기 `EstimateConfig`에 항목이 없으므로,  
  - 1단계: 유무만 저장(`stairs_has_corridor` 또는 `stairs_options.corridor`).  
  - 2단계: 필요 시 관리자 설정에 “계단 복도 월 단가” 추가 후 `calcStairs`에 반영.

### 3.4 복도 처리 요약

- **지금**: 견적계산기에는 복도 항목 없음.
- **권장**:  
  - 현장 거래 글쓰기(계단 전용 Step 3)에는 **“복도 청소 있음” 체크(유무)** 를 넣고,  
  - DB에는 `stairs_has_corridor` 또는 `stairs_options.corridor`로만 저장.  
  - 이후 관리자 견적 설정에 “계단 복도 월 단가”를 추가한 뒤, `StairsInput`·`calcStairs`에 `corridor` 필드를 넣어 반영하면 됨.

---

## 4. 정리

- 계단 청소는 **평수 대신** 견적계산기와 같이 **층수, 주 회수, 화장실/분리수거/엘리베이터/주차장/창틀** 등으로 처리하는 것이 맞고,  
  **복도**는 견적계산기에는 없으므로 “유무”만 먼저 넣고, 단가는 나중에 설정·계산 로직에 추가하는 방식이 좋습니다.
- **구현 순서 제안**  
  1. 업무 종류가 “계단청소”일 때만 Step 3를 계단 전용(층수, 주회수, 화장실, 분리수거, 복도, 기타 옵션)으로 분기  
  2. DB에 계단 전용 컬럼 또는 `stairs_options` JSONB 추가  
  3. createListing/편집에서 해당 필드 저장  
  4. (선택) 상세/목록에서 계단 시 “예상 월액” 표시 시 `calcStairs` 재사용  
  5. (선택) 견적 설정·calcStairs에 “복도” 항목 추가

이 문서는 설계·설명용이며, 실제 스키마/UI 변경은 위 순서대로 적용하면 됩니다.
