# 현장거래 UX·판매자 신뢰도 설계

## 1. 배경

- **현장거래**: 거래 당사자가 직접 연락해 거래 성사. 시스템은 매물 노출 + **판매자 신뢰도 확인**에 초점.
- **불편 사항**: 대분류 → 소분류 2단계 선택이 번거로움. 인력구인처럼 한 번에 고를 수 있게 하되, 현장거래 목적에 맞게 설계.
- **필요 장치**: 몇 건 판매 완료했는지, 후기, 사기 이력 없음, 좋은 매물인지 확인 가능해야 함.

---

## 2. 카테고리(업무 종류) 단순화 — 인력구인과 비슷한 로직

### 2.1 현재 구조

- **인력구인**: "어떤 작업인가요?" → **프리셋 버튼**(사무실청소, 어린이집, 유리청소, …) + **기타(직접입력)**. 한 번에 하나 선택.
- **현장거래**: **대분류** 셀렉트 → **소분류** 셀렉트(대분류에 따라 변경) + 기타 시 직접 입력. 2단계.

### 2.2 제안: 현장거래도 "업무 종류" 단일 선택

- **표시**: 인력구인처럼 **버튼형 프리셋** + **기타(직접 입력)**.
- **저장**: 기존 `category_main_id`, `category_sub_id` 유지. 프리셋 선택 시 해당 ID로 매핑해 저장 → 시장 평균·등급 통계는 그대로 사용.
- **현장거래 유형**(정기청소 소개, 일회성 소개, 정기청소 매매, 현장 도급)은 **그대로 유지**. "거래 유형" + "업무 종류"로 정리.
  - 거래 유형: 드롭다운 1개 (기존 `listing_type`).
  - 업무 종류: 프리셋 버튼 + 기타 1개 (대분류/소분류 대체).

### 2.3 현장거래용 업무 종류 프리셋 예시

- 인력구인 `job-type-presets`와 동일한 `categories` 체계 사용 가능.
- 예: 사무실청소, 어린이집청소, 유리청소, 입주청소, 준공청소, 소독, 고소작업, 외벽청소, … + **기타(직접 입력)**.
- 프리셋 정의: `lib/listings/listing-category-presets.ts` (key, label, category_sub_id 또는 subSlug로 main/sub 자동 매핑).

### 2.4 개선점 정리

| 항목 | 변경 전 | 변경 후 |
|------|---------|---------|
| 입력 단계 | 대분류 선택 → 소분류 선택 (2단계) | 업무 종류 1번 선택 (프리셋 or 기타) |
| 시장 통계 | category_main_id, category_sub_id 사용 | 동일 유지 (백엔드 불변) |
| 사용자 인지 | "대분류/소분류가 뭔지" 고민 | "어떤 청소/업무인지"만 선택 |

## 3. 판매자 신뢰도 장치

### 3.1 요구사항 매핑

| 요구 | 목적 | 현재 상태 | 제안 |
|------|------|-----------|------|
| 몇 건 판매 완료 | 실적 가시화 | `seller_metrics.listing_count`(등록 건수)만 있음 | **거래 완료 건수** 별도 집계·표시 |
| 후기 어떤지 | 신뢰도·품질 | `listing_reviews` 있음, `average_review_rating`, `total_review_count` | 상세 페이지에 **후기 목록** 노출 + SellerCard에 요약 유지 |
| 사기꾼 아님 | 사기 방지 | `listing_incidents`(no_show, non_payment 등) | **신고 누적 시** 등급/뱃지 반영, 필요 시 "주의 판매자" 노출 |
| 좋은 매물인지 | 매물 품질 | 시장 대비 단가(MarketComparisonBox), seller_grade | 유지 + 완료율·후기와 함께 **종합 등급** 강조 |

### 3.2 거래 완료 건수

- **정의**: "판매자로써 이 시스템을 통해 거래를 완료한 건수".
- **구현 옵션**
  - **A**: `listings`에서 `status = 'closed'`인 건수로 집계 (단순, 기존 스키마만 사용).
  - **B**: "거래 완료"를 명시하는 별도 테이블/플래그 (예: `listing_deals.closed_at`)로 집계 (실제 성사 건만 셀 수 있음).
- **권장**: 우선 **A**. 등록자가 "마감" 처리한 건수를 "완료 건수"로 노출. 나중에 "실제 거래 완료" 플로우가 생기면 B로 확장.
- **저장**: `seller_metrics`에 `completed_sales_count` 컬럼 추가하고, 주기 집계(또는 listing status 변경 시 트리거)로 갱신. 또는 매번 `listings`에서 `user_id, status='closed'` count 쿼리.

### 3.3 후기

- **저장**: 기존 `listing_reviews` (rating 1~5, comment).
- **노출**:
  - **상세 페이지**: "이 판매자의 후기" 섹션 — 해당 listing이 아닌 **판매자(user_id) 기준** 최근 후기 N건 노출 (다른 매물 후기 포함).
  - **SellerCard**: 이미 `reviewRating`, `reviewCount` 표시 중 → 유지. "후기 N개" 클릭 시 앵커로 후기 목록으로 스크롤.

### 3.4 사기/신고 방지

- **저장**: 기존 `listing_incidents` (no_show, non_payment, mismatch, no_contact, other).
- **정책**:
  - 신고 누적 시 `seller_metrics.incident_report_count` 반영, `seller_grade` 계산 시 가중치 부여 (이미 등급 로직에 반영 가능하도록).
  - 등급이 C/D이거나 incident가 N건 이상이면 SellerCard에 **"주의"** 문구 또는 뱃지 (예: "신고 이력 있음", "거래 시 주의").
- **선택**: 관리자 전용 "사기 신고" 타입 추가 또는 기존 `other`로 처리 후 관리자 검토.

### 3.5 좋은 매물인지

- **시장 대비 단가**: `MarketComparisonBox` 유지 (평균 대비 S/A/B/C/D).
- **판매자 등급**: `seller_grade` (S~N) — 완료율, 후기, 신고 이력 등 종합. `SellerCard`에서 "매우 좋은 사장" ~ "위험 신호" 문구로 이미 노출.
- **정리**: "몇 건 완료 + 후기 + 사기 이력 없음 + 등급"이 한곳에 보이면 "좋은 매물/신뢰할 수 있는 판매자" 판단 가능.

## 4. UI/UX 제안 요약

### 4.1 글쓰기 · 현장 거래 폼

1. **거래 유형**: 기존처럼 드롭다운 (정기청소 소개, 일회성 소개, 정기청소 매매, 현장 도급).
2. **업무 종류**: 대분류/소분류 제거 → **프리셋 버튼** (사무실청소, 어린이집청소, 유리청소, …) + **기타** 선택 시 텍스트 입력.
3. 제목, 일정, 지역, 금액, 연락처, 상세 내용 등은 유지.

### 4.2 현장 거래 상세 페이지 (판매자 신뢰도)

1. **SellerCard 확장** (또는 "판매자 신뢰도" 블록):
   - **거래 완료 N건** (신규 표시).
   - **등급** (S/A/B/C/D/N) + 짧은 해석 문구 (기존).
   - **후기** 평점·건수 + "후기 보기" 링크 (기존 유지, 링크 시 아래 후기 목록으로).
   - **주의 사항**: incident 또는 등급 C/D 이하일 때만 "거래 시 주의" 등 문구.
2. **후기 목록**: 같은 판매자( user_id )의 다른 매물 포함 후기 최근 N건 리스트 (별도 섹션).

### 4.3 목록 카드

- 카드에 **판매자 등급** 또는 **완료 건수** 요약 한 줄 표시하면 "좋은 매물/신뢰 판매자" 스캔에 유리 (이미 `seller_grade` 노출 중이면 완료 건수만 추가 가능).

## 5. 데이터·스키마 변경 요약

| 항목 | 내용 |
|------|------|
| **listings 폼** | 대분류/소분류 셀렉트 제거 → 업무 종류 프리셋 + 기타 (category_main_id, category_sub_id는 서버에서 프리셋으로 매핑) |
| **seller_metrics** | `completed_sales_count` 추가 (선택). 또는 View/쿼리로 `listings(user_id, status=closed)` count 사용 |
| **listing_reviews** | 변경 없음. 상세 페이지에서 판매자 기준 후기 목록 쿼리 추가 |
| **listing_incidents** | 변경 없음. 등급/노출 로직에서 "주의" 조건만 명확히 |
| **categories** | 변경 없음. 프리셋이 기존 categories와 1:1 매핑 |

## 6. 구현 순서 제안

1. **현장거래 업무 종류 프리셋**  
   - `lib/listings/listing-category-presets.ts` 추가 (또는 기존 job-type-presets와 공유).  
   - NewListingForm에서 대분류/소분류 셀렉트 제거, 프리셋 버튼 + 기타 입력으로 교체.  
   - createListing 시 선택값 → category_main_id, category_sub_id 매핑.

2. **거래 완료 건수**  
   - `seller_metrics`에 `completed_sales_count` 추가 또는 목록/상세에서 `listings(status=closed)` count 쿼리.  
   - SellerCard(또는 상세 블록)에 "거래 완료 N건" 표시.

3. **상세 페이지 후기 목록**  
   - 판매자 기준 `listing_reviews` 조회 (해당 listing + 동일 user_id 다른 listing 후기).  
   - "이 판매자의 후기" 섹션 추가.

4. **주의/신고 노출**  
   - `incident_report_count` 또는 등급이 C/D일 때 SellerCard에 "거래 시 주의" 문구/뱃지 노출.

이 순서로 적용하면 "대분류/소분류 불편" 해소와 "판매자 신뢰도 확인" 장치를 단계적으로 맞출 수 있다.
