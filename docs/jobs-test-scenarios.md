# 인력 구인 테스트 시나리오

구인·구직 플랫폼에서 반드시 검증해야 할 **권한**, **데이터 무결성**, **악의적 사용**, **UX** 테스트를 정리한 문서입니다.

---

## 1. 권한 테스트

### 1-1. 다른 사람이 내 글 수정 시도

| 상황 | 기대 결과 | 현재 구현 |
|------|-----------|-----------|
| 다른 사용자가 `/jobs/{id}/edit` 접근 | 접근 차단 | ✅ 편집 페이지에서 `post.user_id !== user.id`이면 `notFound()` → 404 |
| API로 수정 요청 (다른 사람이 updateJobPost 호출) | 403/권한 오류 | ✅ `updateJobPost`에서 글 소유자 검사 후 `{ ok: false, error: "권한이 없습니다." }` 반환 |

**검증 방법:**  
- 사용자 A로 로그인 후 사용자 B의 구인글 `/jobs/{B의 글 id}/edit` 직접 이동 → 404.  
- 사용자 A로 로그인한 상태에서 B의 글 id로 `updateJobPost(B글id, {...})` 호출(브라우저 콘솔/API) → 실패 메시지.

---

### 1-2. 다른 사람이 지원자 확정 시도

| 상황 | 기대 결과 | 현재 구현 |
|------|-----------|-----------|
| 다른 사용자가 확정 버튼 호출 (다른 사람 글의 지원자에게 확정) | 실패 | ✅ `confirmApplication`에서 `post.user_id !== user.id`이면 `{ ok: false, error: "권한이 없습니다." }` |
| API 직접 호출 (confirmApplication with 다른 사람 글) | 권한 오류 | ✅ 동일 검사로 실패 |

**검증 방법:**  
- 구인자 A의 글에 지원자 X가 있음. 사용자 B로 로그인 후 해당 글 상세에서 X에게 “확정하기” 클릭 → 실패 메시지.  
- 동일하게 Server Action을 직접 호출해도 권한 오류 반환.

---

### 1-3. 확정된 지원자가 아닌데 현장 비공개 정보 접근

| 상황 | 기대 결과 | 현재 구현 |
|------|-----------|-----------|
| 일반 지원자(미확정)가 private 정보 접근 | 차단 | ✅ `job_post_private_details` RLS: 구인자 또는 **해당 글에 대해 accepted인 지원자**만 SELECT 가능. 미확정 지원자는 조회 불가. |
| URL 직접 접근 (같은 글 상세 페이지) | 차단 | ✅ 상세 페이지에서 `JobPostPrivateDetails`는 `isAcceptedForThisPost && privateDetails`일 때만 렌더. 미확정이면 privateDetails는 RLS로 비어 있음(또는 미노출). |

**참고:** 상세 페이지에서 `job_post_private_details` 조회 시 **인증된 클라이언트**(`createServerSupabase`)를 사용해야 RLS가 올바르게 동작합니다. 익명 클라이언트면 모든 사용자에게 비어 보일 수 있음.

**검증 방법:**  
- A의 글에 B가 지원만 하고 확정 전. B로 로그인 후 `/jobs/{A글id}` 접속 → 상세주소/비공개 안내 등이 보이지 않아야 함.  
- API로 `job_post_private_details` 직접 SELECT 시 B의 세션에서는 해당 글의 row가 없어야 함.

## 2. 데이터 무결성 테스트

### 2-1. 동시 확정 경쟁

| 상황 | 기대 결과 | 현재 구현 |
|------|-----------|-----------|
| 포지션 1명 모집, 구인자가 동시에 2명 확정 | 1명만 성공, 1명 실패 | ✅ `confirmApplication`에서 확정 전에 `acceptedList.length >= required_count` 체크. 두 요청이 거의 동시에 들어와도 DB 트랜잭션/순차 처리에 따라 나중 요청은 “이미 해당 포지션 모집이 마감되었습니다”로 실패하는 것이 기대됨. |

**보강 권장:**  
- 포지션별 “확정” 시 `SELECT ... FOR UPDATE` 또는 유니크 제약 + 재시도로 동시성 제어를 명시적으로 두면 더 안전함.  
- 테스트: 동시에 두 번 확정 요청(스크립트/동시 클릭) → 한 건만 성공, 한 건은 “모집 마감” 메시지.

---

### 2-2. 중복 지원

| 상황 | 기대 결과 | 현재 구현 |
|------|-----------|-----------|
| 동일 user, 동일 position, 한 번 지원 | 1건만 생성 | ✅ `applyToPosition`에서 기존 지원 여부 조회 후 `applied`/`reviewing`/`accepted`면 “이미 지원했습니다” 반환. |
| 동일 user, 동일 position, **동시 요청** | 1건만 생성 | ⚠️ 두 요청이 동시에 들어오면 둘 다 “기존 지원 없음”으로 판단해 INSERT 2번 시도 가능. DB에 `UNIQUE(position_id, user_id)` 있으면 한 건은 제약 위반으로 실패. |

**검증 방법:**  
- 지원하기 버튼 연타(아래 UX 테스트와 연계).  
- DB에 `job_applications (position_id, user_id) UNIQUE` 존재 여부 확인. 있으면 동시 요청 시 한 건만 남고 한 건은 에러 처리 필요.

---

### 2-3. 모집 인원 초과

| 상황 | 기대 결과 | 현재 구현 |
|------|-----------|-----------|
| 모집 3명, 이미 3명 확정 후 추가 확정 시도 | 실패 | ✅ `confirmApplication`에서 `(acceptedList?.length ?? 0) >= posRequired.required_count`이면 “이미 해당 포지션 모집이 마감되었습니다.” 반환. |

**검증 방법:**  
- 3명 모집 포지션에 3명 확정 후, 네 번째 지원자에게 확정 시도 → 실패 메시지.

## 3. 악의적 사용 테스트

### 3-1. 스팸 구인

| 상황 | 기대 결과 | 현재 구현 |
|------|-----------|-----------|
| 동일 사용자가 1분에 3개 초과 글 등록 | 제한 | ✅ 1분 내 3건 이상이면 “1분에 구인글은 3개까지 등록할 수 있습니다.” 반환. |
| 동일 사용자가 10분 내 10개 초과 글 등록 | 제한 | ✅ 10분 내 10건 이상이면 “10분 내 구인글은 10개까지 등록할 수 있습니다.” 반환. |

**검증 방법:**  
- 1분 안에 구인글 4개 등록 시도 → 4번째부터 실패.  
- 10분 안에 11개 등록 시도 → 11번째 실패.

---

### 3-2. 허위 금액

| 상황 | 기대 결과 | 현재 구현 |
|------|-----------|-----------|
| 일당 200만원 이상 입력 (pay_unit=day) | 거부 | ✅ 구인 등록/수정 시 “일당 200만원 이상은 허위 금액으로 등록할 수 없습니다.” 반환. |

**검증 방법:**  
- 포지션 단위가 “일당”일 때 2,000,000원 이상 입력 후 저장 → 실패 메시지.

---

### 3-3. 노쇼 악용

| 상황 | 기대 결과 | 현재 구현 |
|------|-----------|-----------|
| 구인자가 무작위로 노쇼 신고 | 관리자 검토 필요 | ✅ 노쇼 신고는 `job_reports`에 기록되고, 관리자 페이지(`/admin/job-reports`)에서 조회 가능. |
| 동일 구인자가 1분에 노쇼 5건 초과 신고 | 제한 | ✅ 1분 내 5건 이상이면 “노쇼 신고는 1분에 5건까지 가능합니다.” 반환. |
| 동일 구인자가 24시간에 노쇼 20건 초과 신고 | 제한 | ✅ 24시간 내 20건 이상이면 “노쇼 신고는 24시간에 20건까지 가능합니다.” 반환. |

**검증 방법:**  
- 동일 구인자로 1분 안에 노쇼 신고 6번 시도 → 6번째 실패.  
- 24시간 내 21번째 노쇼 신고 시도 → 실패.

## 4. UX 테스트

### 4-1. 지원 버튼 연타

| 상황 | 기대 결과 | 현재 구현 |
|------|-----------|-----------|
| 지원하기 연타 | 1회만 처리 | ✅ `ApplyButton`에서 클릭 시 로딩 상태로 비활성화하고, 성공 시 “지원완료”로 전환. 연타해도 첫 요청 이후 버튼이 비활성화되어 중복 클릭 방지. 서버에서도 중복 지원 시 “이미 지원했습니다” 반환. |

**검증 방법:**  
- 지원하기 더블클릭/연타 → 한 번만 지원되고, 두 번째부터는 비활성화 또는 에러 메시지.

---

### 4-2. 뒤로가기 문제

| 상황 | 기대 결과 | 현재 구현 |
|------|-----------|-----------|
| 지원 후 뒤로가기 | 상태 유지 (지원완료로 보임) | ⚠️ 목록/상세가 서버 컴포넌트이므로 뒤로가기 시 `revalidatePath`로 인해 최신 데이터 재요청. 지원 완료 후 뒤로가기하면 “지원완료” 상태로 다시 그려지는 것이 일반적. 단, 브라우저 bfcache에 따라 이전 스냅샷이 잠깐 보였다가 갱신될 수 있음. |

**권장:**  
- 상세 페이지에서 지원 완료 시 `router.refresh()` 또는 `revalidatePath` 호출로 뒤로가기 후에도 최신 상태가 보이도록 이미 적용되어 있는지 확인.  
- 테스트: 지원 완료 → 뒤로가기 → 목록/상세에서 “지원완료” 또는 “지원한 현장”에 해당 글이 반영되는지 확인.

---

## 5. 개선 사항 요약

- **근무일 검증:** 구인 등록/수정 시 `work_date`가 오늘 이전이면 “근무일은 오늘 이후로 선택해 주세요.” 반환.
- **마감 실패 피드백:** 글 전체 마감 실패 시 `JobPostOwnerActions`에서 에러 메시지 표시(`role="alert"`).
- **비로그인 “내가 지원한 현장”:** `/jobs?mine=applied` 비로그인 접근 시 “로그인하면 지원한 현장을 볼 수 있습니다” + 로그인하기 버튼.
- **로딩 스켈레톤:** `/jobs` 로딩 시 `app/jobs/loading.tsx`로 목록 스켈레톤 표시.
- **스팸 구인 제한:** 1분에 3개 초과·10분 내 10개 초과 시 등록 거부.
- **허위 금액 제한:** 일당 200만원 이상(pay_unit=day) 입력 시 등록/수정 거부.
- **노쇼 악용 제한:** 동일 구인자 1분에 노쇼 5건 초과·24시간에 20건 초과 시 신고 거부.

---

## 6. 테스트 실행 시 체크리스트

- [ ] 권한: 타인 글 수정 페이지 접근 → 404.  
- [ ] 권한: 타인 글 수정 API → 권한 오류.  
- [ ] 권한: 타인 글 지원자 확정 → 권한 오류.  
- [ ] 권한: 미확정 지원자로 상세 접근 시 비공개 정보 미노출.  
- [ ] 무결성: 모집 인원 초과 확정 시도 → 실패.  
- [ ] 무결성: 동시 확정 시 1건만 성공.  
- [ ] 무결성: 중복 지원(연타) 시 1건만 생성.  
- [ ] 악의적: 스팸 구인 제한(1분 3건, 10분 10건).  
- [ ] 악의적: 허위 금액(일당 200만원 이상) 거부.  
- [ ] 악의적: 노쇼 신고 rate limit(1분 5건, 24h 20건) 및 관리자 조회 가능.  
- [ ] UX: 지원 버튼 연타 시 1회만 처리.  
- [ ] UX: 지원 후 뒤로가기 시 상태 유지(갱신) 확인.
