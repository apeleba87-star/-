# 현장거래 평균 단가 보완점 — 적용 정리 및 구현 방안

## 1. 수용·즉시 적용 가능 항목

아래는 총평에서 제안된 보완점 중 **우리 시스템에 바로 적용 가능한 것**만 정리한 것이다.  
목적이 다르거나 데이터/기능이 아직 없는 항목은 §2 예외로 둔다.

---

### 1.1 평균 원천 단계 구분 (등록 → 완료 방향)

**수용 내용**
- 평균을 **등록 평균** / **성사·완료 평균**으로 구분하는 방향으로 설계.
- 당장은 **등록 평균**만 구현 가능. 장기적으로 리스팅에 성사·완료 상태를 두면 **완료 평균** 추가.

**구현**
- `market_benchmarks`(또는 현장거래 전용 집계 테이블)에 **`data_source`** 컬럼: `'registered'`(등록) / 추후 `'completed'`.
- 집계·노출 시 항상 **어떤 기준인지** 표시: "등록 데이터 기준", "성사·완료 기준" 등.
- 문서에 **로드맵** 명시: "1단계 등록 평균 → 2단계(선택) 상호작용 → 3단계 완료 평균(리스팅 상태 도입 후)".

**예외**
- **상호작용 평균**(조회/문의 발생 글 기준): 조회·문의 로그가 없으므로 **구현 보류**. 로그 수집 도입 시 검토.

---

### 1.2 노출 시 신뢰도 정보 함께 표시

**수용 내용**
- 평균만 보여주지 말고 **표본 수**, **최근 반영 시점**, **데이터 기준**, **신뢰도 라벨**을 함께 노출.

**구현**
- **저장**: 집계 테이블에 `sample_count`, `updated_at` 이미 있음. `data_source` 추가 시 "등록 기준" 등 표시.
- **신뢰도 라벨**: 표본 수·기준에 따라 서버 또는 클라이언트에서 계산.
  - 표본 &lt; 5: "참고용"
  - 5 ≤ 표본 &lt; 20: "보통"
  - 표본 ≥ 20: "신뢰 높음"
- **UI**: 글쓰기·상세에서 평균 블록에 아래를 함께 표시.
  - "최근 N건 기준"
  - "등록 데이터 기준"
  - "참고용 지표" / "보통" / "신뢰 높음"

---

### 1.3 중앙값·최소·최대 저장 및 노출

**수용 내용**
- 평균뿐 아니라 **중앙값(median)** 을 저장·노출. 최소·최대·표본 수도 저장해 왜곡을 줄인다.

**구현**
- **스키마**: 집계 테이블에 컬럼 추가  
  `median_value`, `min_value`, `max_value` (유형별 금액 1개 기준이면 1열, 복수 지표면 유형별 컬럼).
- **집계**: `refresh_*` 함수에서 `PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY amount)`, `MIN`, `MAX` 계산.
- **노출**: 글쓰기·상세에서 "평균 〇〇만원 · **중앙값 〇〇만원** · 표본 N건" 형태로, **중앙값을 더 강조** (예: 굵게 또는 먼저).

---

### 1.4 다층 집계 + 표본 부족 시 Fallback

**수용 내용**
- 조건 축이 많으면 표본이 쪼개지므로, **레벨 1(지역+유형+업무) → 2(+평수대) → 3(+주회수) → 4(+난이도)** 로 다층 집계.
- 글쓰기 화면에서는 **가장 세밀한 조건부터 조회**하고, **표본이 부족(예: &lt; 5)하면 상위 레벨로 자동 fallback**.

**구현**
- **저장 방식 (택1)**  
  - (A) 집계 테이블에 `aggregation_level` (1~4)를 두고, 레벨별로 행을 넣어 둠.  
  - (B) 조회 시점에 레벨 4 → 3 → 2 → 1 순으로 쿼리해, `sample_count >= N` 인 첫 결과 사용.
- **권장**: (B)로 하면 스키마 변경을 줄일 수 있음.  
  - 예: `get_listing_benchmark(region, listing_type, category_*, area_band, visits_per_week, difficulty)`  
  - 내부에서 4→3→2→1 순으로 조건을 넣었다 뺐다 하며 조회, 표본 5건 이상인 첫 레벨 반환.
- **Fallback 규칙**: 최소 표본 수(예: 5) 미만이면 한 단계 위 조건으로 재조회. 레벨 1까지 갔는데도 부족하면 "표본 부족으로 참고값 없음" 노출.

---

### 1.5 필수 입력 최소화, 선택 입력 안내

**수용 내용**
- **필수**: 정기소개(지역, 업무, 소개비 또는 성사금액 중 1개), 정기매매(지역, 업무, 월 수금, 매매가), 도급(지역, 업무, 월 도급금).
- **선택**: 평수, 주 회수, 난이도. "입력하면 참고 평균이 더 정확해집니다" 문구로 권유.

**구현**
- 글쓰기 폼에서 **필수/선택** 명확히 구분. 선택 필드에는  
  "선택 입력입니다. 입력하시면 해당 조건의 클린아이덱스 평균 참고값을 더 정확히 보여드립니다." 등 문구 노출.
- 서버 검증: 필수만 `required`, 선택은 nullable. 집계·fallback 시 선택값이 있으면 해당 축으로 조회, 없으면 해당 축 무시하고 상위 레벨로 fallback.

---

### 1.6 배수는 저장이 아닌 계산값

**수용 내용**
- 매매에서 "월 수금의 1~6배"는 **입력이 아니라 계산**.  
  저장은 `monthly_amount`, `deal_amount`만 하고, **배수 = deal_amount / monthly_amount** 로 계산·노출·집계.

**구현**
- **저장**: `listings`에 `monthly_amount`, `deal_amount` (유형별로 사용). 배수 컬럼 없음.
- **계산**: 상세·목록·집계에서 `multiplier = deal_amount / monthly_amount` (monthly_amount &gt; 0일 때).
- **집계**: "평균 배수"는 등록된 매매글의 `deal_amount / monthly_amount` 의 평균·중앙값으로 계산.

---

### 1.7 용어 정의 고정 (라벨·설명 문구)

**수용 내용**
- 월 수금, 매매가, 월 도급금, 소개비 등 **용어를 고정**하고, 필드마다 **설명 문구**를 둬서 오입력·혼동을 줄인다.

**구현**
- 폼·상세·목록에서 아래와 같이 **라벨 + 도움말** 통일 (copy 상수화 권장).

| 필드 | 라벨 | 설명 문구 (도움말) |
|------|------|---------------------|
| 월 수금 | 월 수금 | 원청 계약 기준 매월 실제 받는 금액 |
| 매매가 | 매매가 | 해당 현장을 넘길 때 받는 1회 금액 |
| 월 도급금 | 월 도급금 | 실제 도급 운영자에게 지급되는 월 금액 |
| 소개비 | 소개비 | 성사 시 1회 지급되는 금액 |

- placeholder 예: "예: 3500000", "예: 28000000".

---

### 1.8 참고 평균 UI (관행 안내 없음)

**수용 내용**
- **관행 안내**(10%, 80% 등)는 **사용하지 않는다.**
- **플랫폼 참고 평균/중앙값**만 **강조 박스**로, "클린아이덱스 평균 참고값", "최근 N건 기준" 등과 함께 표시.

**구현**
- 관행 안내 문구는 노출하지 않음.
- 평균: `<section className="rounded-xl border border-blue-200 bg-blue-50/50 p-4">` 등으로 구역을 나누고, "이 조건의 최근 **클린아이덱스 평균** 소개비는 약 〇〇만원 (중앙값 〇〇만원, 최근 N건 기준, 등록 데이터 기준, 참고용 지표)" 형태로만 표시.

---

### 1.9 이상치 제거 규칙 (집계 파이프라인)

**수용 내용**
- 집계 전에 **이상치 필터** 적용: 0 이하 제외, 합리적 범위 밖 제외, (표본 충분 시) 상하위 1~5% 절삭 검토. 표본 적으면 절삭하지 않음.

**구현**
- **refresh** SQL/함수 내부에서:
  - `WHERE amount > 0` (및 유형별로 `monthly_amount > 0` 등).
  - 유형별 **합리적 범위** (예: 소개비 1만원~5000만원, 월 수금/도급 10만원~1억, 매매가 0~10억) 밖은 제외하거나 별도 플래그.
  - 표본 수가 충분할 때(예: 20건 이상) `PERCENTILE_CONT(0.05)` ~ `PERCENTILE_CONT(0.95)` 구간만 사용해 평균/중앙값 계산하는 방식 검토. 표본 적으면 절삭 없이 전체 사용.
- **입력 검증**: 글 등록 시에도 동일 범위로 유효성 검사(서버). 너무 극단적이면 "입력 가능 범위를 벗어났습니다" 등 메시지.

---

### 1.10 "시장 평균" 표현 완화

**수용 내용**
- "시장 평균" 대신 **"클린아이덱스 평균 참고값"**, **"최근 등록 데이터 평균"**, **"플랫폼 내 평균 참고값"** 등으로 표현해, 우리 플랫폼 데이터임을 분명히 한다.

**구현**
- UI copy 전역 교체:
  - "시장 평균" → "클린아이덱스 평균 참고값" 또는 "최근 등록 데이터 평균".
  - 필요 시 작은 글씨로 "(우리 플랫폼 등록 데이터 기준)" 보조 문구.

## 2. 예외·나중 단계로 두는 항목

| 항목 | 이유 |
|------|------|
| **상호작용 평균** (조회/문의 발생 글 기준) | 조회·문의 로그/테이블이 없음. 추후 로그 수집 시 검토. |
| **완료 평균 즉시 적용** | 리스팅에 "성사/완료" 상태·완료 시점이 없음. 스키마·플로우 도입 후 2단계에서 적용. |
| **거래 유형별 상태값** (등록됨/연결됨/성사됨/소개비 지급완료 등) | 설계·문서에 "장기적으로 필요"로만 반영. 실제 스키마·업무 플로우는 별도 단계. |
| **허위/장난 데이터 전용 정책** | 이상치 제거·합리적 범위 검증으로 일부 커버. 별도 신고·관리자 검토는 기존 정책 범위에서 유지. |

---

## 3. 구현 순서 제안 (일부 적용됨)

아래 순서로 적용하면, 기존 설계를 유지하면서 보완점을 단계적으로 넣을 수 있다.

**적용된 구현 (마이그레이션 027, 028 + 글쓰기 폼)**  
- **listings** 확장: `monthly_amount`, `deal_amount`, `area_pyeong`, `visits_per_week`, `difficulty` 추가.  
- **listing_benchmarks** 테이블: `region`, `listing_type`, `category_main_id`, `category_sub_id`, `metric_type`, `data_source`, `sample_count`, `avg_value`, `median_value`, `min_value`, `max_value`.  
- **집계 규칙**: 표본 **30건 이상**만 반영, **최대 80건**으로 중앙값·평균·min·max 계산. `refresh_listing_benchmarks()` 함수로 갱신.  
- **글쓰기 폼**: 유형별 금액 필드(소개비/월 수금·매매가/월 도급금), 선택 입력(평수·주 회수·난이도), 참고값(중앙값, N건) 노출.  
- **참고값 갱신**: DB에 반영하려면 주기적으로 `SELECT refresh_listing_benchmarks();` 실행 필요 (cron 등).

### Phase A — 스키마·집계 기반 (현장거래 전용 확장)

1. **listings 확장**
   - 유형별 금액: `monthly_amount`, `deal_amount` (nullable).  
   - 소개: `pay_amount` = 소개비, 선택 `deal_amount`(성사금액).  
   - 매매: `monthly_amount`, `deal_amount` (배수는 계산).  
   - 도급: `monthly_amount` = 월 도급금.  
   - 선택: `area_pyeong`, `visits_per_week`, `difficulty` (nullable).

2. **집계 테이블**
   - 현장거래 전용 benchmark 테이블 또는 기존 `market_benchmarks` 확장.
   - 컬럼: `listing_type`, `region`, `category_main_id`, `category_sub_id`, `area_band`, `visits_per_week`, `difficulty` (또는 aggregation_level 1~4),  
     `data_source` ('registered'),  
     `sample_count`, `updated_at`,  
     `avg_value`, `median_value`, `min_value`, `max_value` (유형별 금액 1종이면 1세트, 복수 지표면 컬럼 추가).

3. **집계 함수**
   - 이상치 필터 (0 이하, 범위 밖 제외).
   - 유형별로 등록 데이터만 집계.
   - 다층: 레벨 1(지역+유형+업무) ~ 4(+난이도)별로 행 삽입하거나, 조회 시 fallback으로 처리.
   - 평균·중앙값·min·max 계산.

### Phase B — 글쓰기·노출

4. **글쓰기 폼**
   - 유형별 필수/선택 구분. 필수 최소화, 평수/주회수/난이도 선택 + "입력하면 참고 평균이 더 정확해집니다".
   - 용어 정의 라벨·도움말 적용. 배수 입력 제거, 월 수금+매매가만 입력.
   - 플랫폼 참고 평균(강조 박스)만 노출. 관행 안내 없음.

5. **benchmark 조회 API/함수**
   - 파라미터: 지역, 유형, 업무, (선택) 평수대·주회수·난이도.
   - 로직: 레벨 4 → 1 순으로 조회, `sample_count >= 5` 인 첫 결과 반환. 없으면 null 또는 "표본 부족".

6. **글쓰기 화면에 참고값 노출**
   - 유형·지역·업무(·선택 조건)에 맞는 집계 1건 조회.
   - "클린아이덱스 평균 참고값 약 〇〇만원 (중앙값 〇〇만원, 최근 N건 기준, 등록 데이터 기준, 참고용/보통/신뢰 높음)" 형태로 표시.

### Phase C — 정리·고도화

7. **상세·목록**
   - 유형별 단가 문구 + 동일 benchmark로 "참고값 대비 ±n%" 등 (선택).

8. **문서·로드맵**
   - "1단계 등록 평균 → 2단계 완료 평균(리스팅 성사/완료 상태 도입 후)" 명시.
   - 용어 정의, 이상치 규칙, fallback 규칙을 운영 문서에 고정.

## 4. 요약

- **바로 적용**:  
  등록/완료 구분 설계, 신뢰도 정보(표본 수·기준·라벨) 노출, **중앙값+min/max** 저장·노출, **다층 집계+fallback**, **필수 최소화+선택 안내**, **배수 계산값**, **용어 정의**, **관행 안내 없음(평균만 노출)**, **이상치 제거**, **"클린아이덱스 평균 참고값"** 표현.
- **예외/추후**:  
  상호작용 평균, 완료 평균(데이터 나올 때까지), 거래 유형별 상태값(별도 기획 후).
- **구현 순서**:  
  스키마·집계(이상치·다층·중앙값) → 글쓰기 폼(필수/선택·용어·배수 계산) → benchmark 조회·fallback → 글쓰기 화면 참고값 노출(신뢰도 정보 포함) → 상세·목록·문서 정리.

이렇게 하면 총평의 보완점 중 우리 목적에 맞는 부분만 골라 바로 적용할 수 있고, 나머지는 예외·로드맵으로 두어 확장 시 일관되게 가져갈 수 있다.

## 6. 악의적 단가 상향 조작 방지

평균 참고값을 끌어올리기 위해 **허위·고단가 글을 의도적으로 다량 등록**하는 행위를 막기 위해 필요한 방어를 정리한다.

### 6.1 위험 시나리오

- 한 사용자(또는 다수 계정)가 **비정상적으로 높은 금액**의 매매·도급·소개글을 반복 등록.
- 등록 데이터 기반 평균·중앙값이 **일시적으로 크게 상승** → 다른 사용자가 참고할 때 시장을 왜곡된 것으로 인식.
- 목적: 자신의 글을 “평균 이하”로 보이게 하거나, 업계 단가 인상을 유도하는 등.

### 6.2 필요한 방어 수단

아래는 **집계 구조**, **입력·등록 제한**, **탐지·대응** 세 축으로 나눈 것이다.

---

#### (1) 집계 단계에서의 완화 — 이미 설계에 포함된 것

| 수단 | 설명 | 효과 |
|------|------|------|
| **중앙값 강조** | 참고값으로 **평균보다 중앙값**을 더 크게/먼저 노출. | 소수의 극단적 고단가만으로는 중앙값이 크게 움직이지 않음. |
| **이상치 제거** | 집계 전 0 이하·합리적 범위 밖 제외, (표본 충분 시) 상하위 1~5% 절삭. | 단일 극단값이 평균을 지나치게 끌어올리는 것 방지. |
| **합리적 범위 검증** | 유형별 상한(예: 월 수금/도급/매매가/소개비 상한) 밖은 집계에서 제외 또는 입력 단계에서 거부. | “9999만원” 같은 허위 고단가가 집계에 들어가지 않음. |
| **표본 수·신뢰도 표시** | “최근 N건 기준”, “참고용” 라벨 노출. | 표본이 적거나 이상이 있으면 사용자가 참고값을 맹신하지 않음. |

→ **이미 §1.3, §1.9, §1.2에 반영.** 구현 시 이상치 규칙과 범위를 엄격히 적용하는 것이 조작 완화의 첫 단계다.

---

#### (2) 입력·등록 단계 제한

| 수단 | 설명 | 구현 방향 |
|------|------|------------|
| **금액 상한(유형별)** | 소개비·월 수금·매매가·월 도급금에 **최대 허용값** 정의. 초과 시 저장 거부 또는 “검토 대상” 처리. | 서버 검증 + (선택) 관리자 검토 플로우. |
| **등록 빈도 제한** | 동일 사용자가 **단기간에 현장거래 글을 과도하게** 올리지 못하도록 제한. (예: 1분에 3건, 24시간에 20건 등 — 인력 구인 스팸 제한과 유사.) | 글 등록 API/액션에서 `user_id` 기준으로 최근 등록 건수 조회 후 제한. |
| **동일 유형·조건 중복 제한** | 같은 사용자가 **동일 지역·업무·유형**으로 비슷한 금액의 글을 매우 많이 올리는 패턴 제한. (선택, 표본 많아진 뒤 검토.) | 예: (user_id, region, listing_type, category)별 최근 N건 제한 또는 “의심” 플래그. |

→ **금액 상한**과 **등록 빈도 제한**을 먼저 적용하는 것을 권장한다.

---

#### (3) 탐지·사후 대응 (운영·장기)

| 수단 | 설명 | 구현 방향 |
|------|------|------------|
| **집계 제외 플래그** | 특정 리스팅이 허위·조작으로 확인되면 **해당 글만 집계에서 제외**하는 플래그. | `listings`에 `exclude_from_benchmark BOOLEAN` 등. refresh 시 `WHERE exclude_from_benchmark IS NOT TRUE`. |
| **관리자 조회** | “특정 지역·유형에서 평균이 비정상 상승한 구간” 또는 “한 사용자의 등록 비중이 과도한 구간” 등을 **대시보드·리포트**로 조회. | 집계 결과·시간 추이, user_id별 기여도 요약 등. |
| **신고·검토** | 사용자 신고 또는 시스템 플래그에 따라 **관리자가 해당 글·사용자**를 검토하고, 조작 시 집계 제외·경고·제재. | 기존 신고/관리자 도구와 연동. |

→ **즉시 구현**보다는, **집계·입력 방어를 먼저** 넣고, 운영 데이터가 쌓이면 탐지·제외 플로우를 단계적으로 도입하는 것이 좋다.

---

### 6.3 구현 우선순위 제안

1. **1차 (집계·표시)**  
   이상치 제거·합리적 범위 적용, **중앙값 저장·노출**, 표본 수·신뢰도 라벨 표시.  
   → 이미 보완 설계에 포함. **반드시 적용.**

2. **2차 (입력·등록)**  
   - 유형별 **금액 상한** 검증 (서버).  
   - 현장거래 글 **등록 빈도 제한** (사용자당 N건/분, N건/일).  
   → 악의적 다량 등록과 극단적 고단가 입력을 동시에 완화.

3. **3차 (운영)**  
   - 집계 제외 플래그, 관리자용 “기여도·이상 구간” 조회, 신고·검토 연동.  
   → 조작 사례가 보이기 시작할 때 단계적으로 도입.

### 6.4 요약

- **평균 단가를 악의적으로 끌어올리려는 행위**를 막으려면  
  **집계 단계(중앙값·이상치·범위)** + **입력 단계(금액 상한·등록 빈도 제한)** 가 필수이고,  
  **탐지·집계 제외·관리자 검토**는 운영이 쌓인 뒤 보강하는 것이 적절하다.
- 이미 설계된 **중앙값 강조·이상치 제거·신뢰도 표시**만으로도 단순 조작의 영향은 크게 줄어든다.  
  여기에 **금액 상한**과 **등록 빈도 제한**을 더하면, 현장거래 평균 참고값을 올리기 위한 악의적 등록을 실질적으로 억제할 수 있다.
